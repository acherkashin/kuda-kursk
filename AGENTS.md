# Границы
## Обязательно
- Markdown-файлы, кроме папки .specify/ и служебной папки .agents/skills/, должны быть на русском
- Markdown-файлы в `specs/` должны быть на русском, но в них разрешены служебные термины Spec Kit и технические/product labels на английском, включая `Clarifications`, `Session`, `Phase`, `Priority`, `Checkpoint`, `User Story`, `MVP`, `Setup`, `Foundational`, `Polish`, `contracts`, `quickstart`, `route`, `slug`
- Markdown-файлы в папке .specify/ должны быть на английском
- Markdown-файлы в папке .agents/skills/ являются служебными инструкциями локальных навыков и не подпадают под языковое правило проекта

# Процесс
- Агент не должен создавать новую ветку без явного предварительного подтверждения пользователя.
- Исключение: если пользователь явно запускает или просит использовать `speckit` / Spec Kit workflow, Spec Kit может создавать или переключать feature branch без дополнительного подтверждения.

## Spec Kit
- В проекте используется [Spec Kit](https://github.com/github/spec-kit).
- Текущая фича, над которой мы работаем, определяется именем активной ветки; 
- Описание фич хранится в папке specs.

## Версионирование
- После завершения очередной фичи инкрементируйте `package.json.version` вручную по SemVer.
- Используйте `patch` для исправлений и мелких улучшений без нового пользовательского сценария.
- Используйте `minor` для новой пользовательской функции или заметного расширения поведения.
- Используйте `major` для несовместимых изменений публичного поведения, данных или процесса.

## Обязательное чтение
Перед планированием или реализацией функции прочитайте:

- `.specify/memory/constitution.md`

Перед планированием или реализацией интерфейса явно прочитайте:

- `DESIGN_SYSTEM.md`

При любых изменениях дизайна или интерфейса используйте скилл `ui-ux-pro-max`.

## Синхронизация документации
После внесения изменений в код обновляйте следующие файлы текущей фичи, чтобы код и требования оставались согласованными:
- `specs/<feature>/spec.md` — если изменилось поведение, scope или требования
- `specs/<feature>/plan.md` — если изменился подход, шаги реализации или статус задач
- `specs/<feature>/tasks.md` — отмечайте выполненные задачи и добавляйте новые, выявленные в процессе работы

## Тесты
- Новые или изменённые unit/e2e/integration тесты добавляются только после явного запроса или разрешения пользователя.
- Если агент считает новое тестовое покрытие полезным, он сначала кратко объясняет риск и запрашивает разрешение.
- Без отдельного разрешения используйте существующие тесты, typecheck, build, ручную проверку, visual QA или другие уже согласованные проверки, если они уместны.
- Визуальная стилистика не должна покрываться автоматическими тестами: не фиксируйте в тестах цвета, палитру, hex/rgb/hsl-значения, CSS-переменные цвета, цветовые Tailwind-классы, шрифты, размеры шрифтов, отступы, радиусы, тени, точную геометрию layout, пересечения элементов, screenshots/snapshots и декоративные CSS-классы.
- Автоматические тесты интерфейса должны проверять поведение, доступность, маршруты, состояние, данные и продуктовые контракты, а не внешний вид.

## Storybook
- Названия Storybook stories должны быть на английском: используйте английские `title` segments и именованные exports в `*.stories.tsx`.
- Русский текст внутри самих компонентов, mock-данных и пользовательского контента в stories разрешён и должен соответствовать продуктовой локали.
- При добавлении или изменении значимого интерфейсного состояния добавляйте или обновляйте Storybook story для этого состояния, если компонент уже представлен в Storybook или новое состояние важно для визуальной проверки.

## Аналитика
- При реализации нового пользовательского сценария или значимого действия пользователя добавляйте событие Яндекс.Метрики через typed analytics adapter.
- Не отправляйте персональные данные, сырые поисковые запросы, произвольные URL, user agent и нестабильные технические детали.
- Новые события синхронизируйте с `src/domain/analyticsEvents.ts` и `specs/<feature>/contracts/analytics-events.md`.
- Если событие спорное по приватности, продуктовой ценности или гранулярности, сначала кратко опишите риск и запросите решение пользователя.

# Компонентный подход и дизайн-система

## Обязательные правила
- **Адаптивность обязательна.** Все изменения UI должны корректно отображаться и оставаться удобными как на десктопных, так и на мобильных экранах. Перед завершением работы необходимо провести visual QA обоих вариантов.
- **Запрещено копипастить стили.** Один и тот же визуальный паттерн (кнопка, иконка-кнопка, ссылка) должен быть вынесен в разделяемый компонент — не дублировать `className` в нескольких файлах.
- **Разделяемые UI-примитивы** живут в `src/components/ui/` и служат единственным источником правды для оформления интерактивных элементов.
- **Перед добавлением нового inline-стиля** убедитесь, что в `src/components/ui/` нет подходящего компонента. Если подходит с небольшими вариациями — расширьте существующий через пропсы.
- **Каждый новый повторяемый паттерн** (который встречается дважды и более) извлекается в `src/components/ui/` сразу при его появлении.

## Текущие UI-примитивы
| Компонент | Файл | Назначение |
|-----------|------|------------|
| `IconButton` | `src/components/ui/IconButton.tsx` | Круглая кнопка-иконка (закрыть, назад, поиск, о проекте) |
| `Button` | `src/components/ui/Button.tsx` | Кнопка с текстом (primary, secondary, accent, accent-soft) |
| `ButtonLink` | `src/components/ui/ButtonLink.tsx` | Ссылка `<a>` со стилями кнопки |

# Технологии
- Frontend-only PWA на React, Vite и TypeScript.
- Tailwind CSS v4 для стилизации интерфейса.
- Стили необходимо писать через утилиты Tailwind; нативный CSS использовать только когда задачу нельзя корректно решить средствами Tailwind.
- Motion для анимаций и переходов состояния.
- MapLibre GL JS с OpenFreeMap для карты.
- vite-plugin-pwa и workbox-window для PWA-слоя.
- React Router Declarative Mode для маршрутизации.
- Собственные TypeScript-валидаторы для данных, без Zod.
- Яндекс.Метрика подключается только после явного согласия пользователя.
