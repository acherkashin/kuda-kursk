# Быстрый старт: Куда в Курске

## Подготовка

1. Установить зависимости frontend-проекта:

```bash
pnpm install
```

2. Для локальной проверки Метрики задать ID счётчика только в `.env.local`:

```bash
VITE_YANDEX_METRIKA_ID=00000000
```

Если переменная не задана, analytics adapter должен работать в no-op режиме.

## Локальный запуск

```bash
pnpm dev
```

Ожидаемо: открывается SPA с полноэкранной картой Курска, маркерами и поиском. До согласия пользователя script Яндекс.Метрики не загружается.

## Проверки качества

```bash
pnpm typecheck
pnpm test:unit
pnpm test:e2e
pnpm build
```

Ожидаемо: TypeScript проходит без ошибок; существующие unit-проверки проверяют поиск, координаты и route links; существующие Playwright-сценарии проходят desktop/mobile/PWA/analytics проверки; production build создаёт PWA manifest и service worker.

Новые или изменённые тесты не добавляются в рамках проверки качества без отдельного явного запроса или разрешения пользователя; существующие команды выше можно запускать как регрессионную проверку.

В GitHub Actions тот же регрессионный набор запускается workflow `PR checks` на pull request, ручной `workflow_dispatch` и каждый push в ветку `main`; шаг `E2E tests` выполняет существующий `pnpm test:e2e` и при падении сохраняет Playwright report artifact.

## Проверка PWA

1. Собрать production build:

```bash
pnpm build
pnpm preview
```

2. Открыть preview в Chromium.
3. Проверить manifest, icons 192/512, единственную service worker registration и offline app shell; в `dist` не должно быть `registerSW.js`, а `index.html` не должен подключать дополнительный registration script.
4. Последовательно открыть две версии production-сборки на одном origin и убедиться, что update prompt не появляется, новая версия service worker получает контроль над вкладкой, страница автоматически перезагружается ровно один раз при update-событии, а первичная установка worker не вызывает перезагрузку.
5. При доступной сети убедиться, что `/data/*.json` возвращает свежие данные и обновляет runtime cache; затем отключить сеть и проверить fallback на последнюю кешированную версию.
6. Убедиться, что внешние raster tiles CARTO Positron не кешируются без ограничений и что приложение корректно показывает резервное состояние при недоступной карте.
7. На Android Chrome проверить мобильную подсказку установки и native prompt по кнопке «Установить».
8. На iPhone Safari проверить инструкцию «Поделиться» → «На экран “Домой”» → «Добавить» и отсутствие install CTA и install-секции в «О проекте» после открытия установленной PWA.

## Проверка аналитики

1. Открыть приложение в новом профиле браузера.
2. До выбора в cookie/analytics banner убедиться, что:
   - `window.ym` отсутствует;
   - запрос к script Метрики не выполнялся;
   - analytics cookies/localStorage Метрики не появились.
3. Нажать "Принять аналитику".
4. Убедиться, что script Метрики загружен и события проходят через typed adapter.
5. Выполнить поиск и проверить, что событие `search_used` не содержит сырой текст запроса.
6. Выполнить сценарий установки PWA и убедиться, что install-события отправляются только после нажатия «Установить» / «Показать шаги», закрытия install-плашки или browser install-события; параметры содержат только `platform`, `source` и безопасный `outcome`, без URL, user agent и персональных данных.
7. Отозвать согласие через настройки аналитики и убедиться, что будущие события не отправляются.

## Проверка карт сообществ

1. Открыть `/maps/<known-slug>` и убедиться, что отображается отдельная идентичность карты и нужный набор мест.
2. Открыть `/maps/unknown-slug` и убедиться, что приложение показывает понятное fallback-состояние и путь назад на основную карту.
